# AutoReviewer Workbench

Rewrite of AutoReviewer from VBA/Word-COM to a single self-contained HTML tool.
The legacy `.bas` modules in this repo are the *behavioral* reference (flow semantics,
step order, review conventions) — port the flows, never the code.

## Read before writing any code

1. `docs/spec-workbench.md` — the implementation spec. Authoritative for pipeline behavior.
2. `docs/spec-stickshift-compliance.md` — packaging/integration standard. Authoritative for
   the built artifact; dist MUST pass its §11 checklist.
3. `reference/redline-to-markdown.html` — working docx→CriticMarkup exporter. Port it into
   `src/ooxml/export.js`; do not rewrite from scratch.

## Non-negotiables

- One deliverable: `html/autoreviewer-workbench.html`, committed to the repo and CI-verified
  fresh against source on every PR (see Commands). `dist/autoreviewer-workbench.html` is a
  gitignored local build for quick iteration only, not the shipped artifact. Fully offline,
  runs from `file://`, no external `src`/`href`, no `fetch`, no localStorage/sessionStorage.
- All document I/O client-side: file picker/drag-drop in, Blob download out.
- Clipboard: hidden-textarea `execCommand('copy')` fallback is the PRIMARY path
  (async clipboard API fails on `file://`).
- Validation gate G2 is sacred: strip(CriticMarkup response) must byte-equal the exported
  markdown, or the pipeline blocks. Never weaken it to fuzzy matching.
- Never pretty-print or reformat OOXML. Untouched ZIP parts are copied byte-for-byte.
- Tracked-change attribution: `w:author` default "AutoReviewer — {persona}".
- No edit reaches the .docx without passing the ratification UI (per-edit accept/reject).

## Working agreement

- Work milestone by milestone (spec §15). Propose a plan before implementing each milestone;
  wait for approval. One commit (or PR) per milestone, tests green before moving on.
- `src/zip` and `src/ooxml` must run in Node (Vitest) — no direct DOM-UI imports;
  DOMParser/XMLSerializer injected per environment.
- Fixtures are generated by `fixtures/generate.py` (python-docx) and committed; document
  every fixture's construction in that script.

## Commands

- `npm test` — Vitest
- `npm run build` — bundle + inline into `dist/autoreviewer-workbench.html` (local, gitignored)
- `npm run build:html` — same, but into the committed `html/autoreviewer-workbench.html`.
  Rebuild and recommit this file as part of any PR touching `src/`, `template.html`, or
  `build/` — CI fails the build if it's stale relative to source.
- `npm run check:compliance [path]` — static checks from compliance standard §11 (defaults
  to `dist/`; CI passes `html/autoreviewer-workbench.html`)
- After a milestone's PR merges to `main`, push a tag (e.g. `m3a`) to trigger the release
  workflow, which attaches the committed `html/autoreviewer-workbench.html` to a GitHub
  Release named for the tag.
